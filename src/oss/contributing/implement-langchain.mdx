Integration packages are Python packages that users can install for use in their projects. They implement one or more components that adhere to the LangChain interface standards.

LangChain components are subclasses of base classes in [`langchain-core`](https://github.com/langchain-ai/langchain/tree/master/libs/core). Examples include [chat models](/oss/integrations/chat), [tools](/oss/integrations/tools), [retrievers](/oss/integrations/retrievers), and more.

Your integration package will typically implement a subclass of at least one of these components. Expand the tabs below to see details on each.

    <Tab title="Tools">
        Tools are used in 2 main ways:

        1. To define an "input schema" or "args schema" to pass to a chat model's tool calling feature along with a text request, such that the chat model can generate a "tool call", or parameters to call the tool with.
        2. To take a "tool call" as generated above, and take some action and return a response that can be passed back to the chat model as a ToolMessage.

        The Tools class must inherit from the @[`BaseTool`] base class. This interface has 3 properties and 2 methods that should be implemented in a subclass.

    <Tab title="Checkpointers">
        Checkpointers enable [persistence](/oss/langgraph/persistence) in LangGraph, allowing agents to save and resume state across interactions. Checkpointer integrations subclass @[`BaseCheckpointSaver`] from `langgraph-checkpoint` and persist LangGraph checkpoint records, channel blobs, metadata, and intermediate writes in an external backend.

        ## Implement the storage contract

        A production checkpointer should implement the synchronous and asynchronous forms of the core saver methods:

        | Method | Purpose |
        |--------|---------|
        | `put` / `aput` | Store a checkpoint, its metadata, and any channel blobs referenced by `new_versions`. Return a config containing `thread_id`, `checkpoint_ns`, and the stored `checkpoint_id`. |
        | `get_tuple` / `aget_tuple` | Fetch a checkpoint tuple by config. Include the checkpoint, metadata, parent config, and pending writes for the checkpoint. |
        | `list` / `alist` | List checkpoints for a thread or namespace, with support for `filter`, `before`, and `limit`. |
        | `put_writes` / `aput_writes` | Store intermediate writes keyed by checkpoint, task ID, and write index. Use `WRITES_IDX_MAP` from `langgraph.checkpoint.base` for special channels such as errors, interrupts, and resumes. |
        | `delete_thread` / `adelete_thread` | Delete all checkpoints, blobs, and writes for a thread. |

        Optional Agent Server operations include `delete_for_runs` / `adelete_for_runs`, `copy_thread` / `acopy_thread`, and `prune` / `aprune`.

        Model your backend around these logical records:

        - A checkpoint record keyed by `thread_id`, `checkpoint_ns`, and `checkpoint_id`, plus `parent_checkpoint_id`, serialized checkpoint data, and metadata.
        - Channel blob records keyed by `thread_id`, `checkpoint_ns`, channel name, and channel version. Only store blobs for channel versions provided in `new_versions`; preserve empty markers for versions without a value.
        - Write records keyed by `thread_id`, `checkpoint_ns`, `checkpoint_id`, `task_id`, and write index. Store the channel, serialized value, and `task_path` so pending writes can be returned in deterministic order.

        ## Add delta channel support

        `DeltaChannel` reduces checkpoint storage by storing a delta sentinel instead of the full channel value on most steps. LangGraph reconstructs the channel by replaying ancestor writes through the channel reducer. A checkpointer supports delta channels when it can answer that reconstruction query correctly.

        To support delta channels:

        1. Preserve delta sentinel and snapshot values exactly through your serializer. Do not drop unknown sentinel-like values from `checkpoint["channel_values"]` or channel blob storage.
        2. Store writes through `put_writes` / `aput_writes` before the next checkpoint is committed, and return those writes from `get_tuple` as `pending_writes` in stable oldest-to-newest order.
        3. Preserve `parent_checkpoint_id` and return `parent_config` from `get_tuple`. Delta reconstruction follows the parent chain, not a timestamp-ordered `list()` result, so forked threads must only replay writes on the selected path.
        4. Implement `_get_channel_writes_history(config, channel)` and `_aget_channel_writes_history(config, channel)` if your backend can query storage directly. The method should start at the target checkpoint's parent, exclude writes stored at the target checkpoint, collect writes for the requested channel oldest-to-newest, and stop at the nearest non-sentinel blob. The returned seed is that blob value; if none exists, return `DELTA_SENTINEL` as the seed.
        5. Treat `_DeltaSnapshot` blobs as seed values. Delta channels with `snapshot_frequency` write these snapshots periodically to bound replay depth.
        6. Preserve pre-delta migration behavior. If a thread was written before a channel switched to `DeltaChannel`, the nearest plain stored value for that channel is the seed and older writes must not be replayed past it.

        The default `_get_channel_writes_history` implementation can work for simple savers whose `get_tuple` returns complete `parent_config` and `pending_writes` data. Backends with direct storage access should override it for performance, following the in-memory and PostgreSQL implementations in the LangGraph repo.

        ## Test your checkpointer

        Validate the core contract with the conformance suite from `langgraph-checkpoint-conformance`:

        ```bash
        pip install langgraph-checkpoint-conformance
        ```

        The suite checks required capabilities such as `put`, `put_writes`, `get_tuple`, `list`, and `delete_thread`, and auto-detects optional operations such as `copy_thread` and `prune`. Add targeted tests for delta-channel behavior until your backend is covered by the conformance suite: fresh delta threads, migration from an existing reducer such as `add_messages`, fork/time-travel replay, snapshot frequency, and sync plus async saver paths.

        ## Reference implementations

        Start with the [in-memory, SQLite, and PostgreSQL savers](https://github.com/langchain-ai/langgraph/tree/main/libs) in the LangGraph repo. The [checkpointer integrations page](/oss/integrations/checkpointers) links additional backend packages, including AWS, MongoDB, Azure Cosmos DB, Redis, CockroachDB, and Aerospike implementations.

    <Tab title="Sandboxes">
        Sandbox integrations enable [Deep Agents](/oss/deepagents/overview) to run code in isolated environments.

        :::python
        Implement the @[`SandboxBackendProtocol`] from Deep Agents. This protocol includes `execute()`, async variants, and the filesystem tool methods such as `ls`, `read`, `write`, `edit`, `glob`, and `grep`.

        In practice, if your sandbox environment can run shell commands and has `python3` available, you should usually subclass @[`BaseSandbox`]. `BaseSandbox` provides the filesystem operations through `python3`, so you mainly need to implement `execute()`, `upload_files()`, `download_files()`, and `id`.

        ```python Example BaseSandbox scaffold expandable
        from __future__ import annotations

        from deepagents.backends.protocol import (
            ExecuteResponse,
            FileDownloadResponse,
            FileUploadResponse,
        )
        from deepagents.backends.sandbox import BaseSandbox  # [!code highlight]


        class MySandbox(BaseSandbox):
            def __init__(self, client: MySandboxSdkClient) -> None:
                self._client = client

            @property
            def id(self) -> str:
                return self._client.sandbox_id

            def execute(
                self,
                command: str,
                *,
                timeout: int | None = None,
            ) -> ExecuteResponse:
                # Execute `command` in your sandbox and map the provider response
                # into ExecuteResponse.
                result = self._client.run(command=command, timeout=timeout)
                output = result.stdout or ""
                if result.stderr:
                    output += f"\n<stderr>{result.stderr}</stderr>"
                return ExecuteResponse(
                    output=output,
                    exit_code=result.exit_code,
                    truncated=False,
                )

            def upload_files(
                self,
                files: list[tuple[str, bytes]],
            ) -> list[FileUploadResponse]:
                # Validate paths, batch requests where possible, and map provider
                # results back into FileUploadResponse objects in input order.
                # Only catch and normalize errors that an LLM can plausibly retry
                # or fix, such as invalid_path or file_not_found.
                return self._client.upload_files(files)

            def download_files(self, paths: list[str]) -> list[FileDownloadResponse]:
                # Validate paths, batch requests where possible, and map provider
                # results back into FileDownloadResponse objects in input order.
                # Only catch and normalize errors that an LLM can plausibly retry
                # or fix, such as invalid_path or file_not_found.
                return self._client.download_files(paths)

            async def aexecute(
                self,
                command: str,
                *,
                timeout: int | None = None,
            ) -> ExecuteResponse:
                ...

            async def aupload_files(
                self,
                files: list[tuple[str, bytes]],
            ) -> list[FileUploadResponse]:
                ...

            async def adownload_files(
                self,
                paths: list[str],
            ) -> list[FileDownloadResponse]:
                ...
        ```

        ## Test your integration

        Validate your integration with the [sandbox standard test suite](/oss/contributing/standard-tests-langchain#sandbox-integrations). The Python suite uses `SandboxIntegrationTests` from `langchain_tests.integration_tests`; subclass it and provide a `sandbox` fixture that yields a clean `SandboxBackendProtocol` instance.

        ```python Example sandbox standard test setup expandable
        from __future__ import annotations

        from collections.abc import Iterator

        import pytest
        from deepagents.backends.protocol import SandboxBackendProtocol
        from langchain_tests.integration_tests import SandboxIntegrationTests

        from langchain_myprovider import MySandbox
        from myprovider_sdk import MySandboxSdkClient


        class TestMySandboxStandard(SandboxIntegrationTests):
            @pytest.fixture(scope="class")
            def sandbox(self) -> Iterator[SandboxBackendProtocol]:
                client = MySandboxSdkClient()
                backend = MySandbox(client=client)
                try:
                    yield backend
                finally:
                    # Replace this with your provider's cleanup logic.
                    client.delete_sandbox(backend.id)
        ```

        Put this in a file such as `tests/integration_tests/test_sandbox.py`. The standard suite will handle the actual filesystem and command-execution assertions for you.

        **Reference implementation:** See the [Daytona partner integration](https://github.com/langchain-ai/deepagents/tree/main/libs/partners/daytona), which subclasses `BaseSandbox` and implements `execute()`, `upload_files()`, `download_files()`, and `id`.
