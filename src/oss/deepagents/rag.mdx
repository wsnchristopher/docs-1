This guide shows how to build a [Retrieval Augmented Generation (RAG)](/oss/langchain/retrieval/) application with [Deep Agents](/oss/deepagents) instead of a single LangChain agent loop. The pattern mirrors the [LangChain RAG tutorial](/oss/langchain/rag), but uses Deep Agents capabilities to keep the orchestrator context clean:

1. **Index** LangChain documentation into a vector store.
2. **Search** with a custom tool that runs vector similarity search and writes each retrieved chunk to the agent [filesystem](/oss/deepagents/backends).
3. **Analyze** by delegating each chunk file to a [subagent](/oss/deepagents/subagents) that reads the file and returns a focused summary.
4. **Synthesize** a final answer in the main agent from subagent reports.

This tutorial uses [docs.langchain.com](https://docs.langchain.com) as the corpus. For a simpler RAG agent without subagents or filesystem offloading, see [Build a RAG agent with LangChain](/oss/langchain/rag).

### Key concepts

This tutorial covers:

- [Vector indexing](/oss/langchain/rag#index-your-content) with LangChain document loaders and embeddings
- Custom [tools](/oss/langchain/tools) that update agent state with @[`Command`]
- [Subagents](/oss/deepagents/subagents) for context-isolated chunk analysis
- The default [StateBackend](/oss/deepagents/backends#statebackend) filesystem for retrieved chunks

## Prerequisites

API keys for:

- A chat model provider (Anthropic, OpenAI, or Google)
- OpenAI (or another [embeddings integration](/oss/integrations/embeddings)) for indexing
- [LangSmith](https://smith.langchain.com) for tracing (optional)

## Index LangChain documentation

In the indexing step, fetch documentation pages, split them into chunks, embed the chunks, and store them in a `VectorStore`. The agent searches this index at runtime; it does not re-fetch the full site on every question.

Answer questions about LangChain using the indexed documentation corpus.

1. **Plan**: Use write_todos to break complex questions into focused search queries.

4. **Synthesize**: Combine subagent summaries into a final answer with inline links to documentation sources.
5. **Verify**: If summaries do not fully answer the question, run another search with a refined query.

Do not answer from memory when documentation evidence is required. Search first.

When the agent runs, it typically:

1. Calls `search_documentation` with a query about retrieval tools or subagents.
2. Receives file paths such as `/retrieved/a1b2c3d4/chunk_1.md`.
3. Launches one or more `task()` calls to `chunk-analyst`, each scoped to a single chunk file.
4. Synthesizes a final answer with links to the relevant documentation pages.

If you enabled LangSmith in [Setup](#setup), open [LangSmith](https://smith.langchain.com) and inspect the trace to see search calls, filesystem writes, subagent delegations, and the final response.

## Security considerations

<Warning>
RAG applications are susceptible to **indirect prompt injection**. Retrieved documentation may contain text that resembles instructions. Because retrieved chunks share the context window with your system prompt, models may follow instructions embedded in documentation rather than your intended prompt.
</Warning>

To mitigate this:

1. **Use defensive prompts**: Instruct the orchestrator and subagents to treat retrieved content as data only, as shown in the prompts above.
2. **Wrap context with delimiters**: The search tool stores chunks with a `# Source:` header so analysts can distinguish metadata from body content.
3. **Validate responses**: Check that answers cite expected documentation paths and reject unexpected output formats when integrating into a product UI.

For more on this topic, see the [security section](/oss/langchain/rag#security-considerations) of the LangChain RAG tutorial and research on [prompt injection](https://simonwillison.net/series/prompt-injection/).
