This page covers two teardown scenarios for deployments provisioned with the [GCP Terraform modules](https://github.com/langchain-ai/terraform/tree/main/modules/gcp):

1. **With Terraform state** — the happy path using `terraform destroy`.
2. **Without Terraform state** — manual cleanup with the gcloud CLI when Terraform state isn't available.

Both follow the same reverse-dependency order. Pick the section that matches your situation.

<Warning>
Teardown permanently deletes the Cloud SQL instance, GCS bucket contents, and Secret Manager secrets. Export any data you need to keep before proceeding.
</Warning>

<Tip>
Before destroying, review the [LangSmith self-hosted changelog](/langsmith/self-hosted-changelog) for data migration or export notes.
</Tip>

## Pre-teardown checklist

```bash
# Re-auth if Terraform or gcloud shows:
# oauth2: "invalid_grant" "reauth related error (invalid_rapt)"
gcloud auth login
gcloud auth application-default login

# Verify GCP identity and active project
gcloud auth list
gcloud config get-value project

# Cluster credentials
gcloud container clusters get-credentials <cluster-name> --region <region> --project <project-id>

# Confirm kubectl points to the right cluster
kubectl config current-context

# Check what is running
helm list -A
kubectl get namespaces
```

## Option A — Teardown with Terraform state

Use this when `terraform state list` returns resources. Teardown happens in reverse order of deployment:

```txt
Pass 3 (if enabled)  → Remove LangGraph deployments (LGP CRDs + pods)
Pass 2               → Uninstall LangSmith Helm release
Pass 1               → Destroy all GCP infrastructure (terraform destroy)
```

### A1 — Remove LangSmith Deployment resources

Skip if Pass 3 was never enabled. The uninstall script in A2 cleans up LGP pods, services, and deployments automatically. The LGP CRD is kept by a Helm `resource-policy: keep` annotation and must be deleted manually. Without manual deletion, the CRD persists indefinitely and causes confusing `kubectl get lgp` results after reinstall.

```bash
kubectl delete lgp --all -n langsmith
kubectl get pods -n langsmith -w   # wait for the operator to finish
kubectl delete crd lgps.apps.langchain.ai
```

### A2 — Uninstall the Helm release

```bash
cd terraform/modules/gcp
make uninstall
```

Or manually:

```bash
helm uninstall langsmith -n langsmith
kubectl get pods -n langsmith
```

After uninstalling:

```bash
kubectl delete namespace langsmith
```

### A3 — Remove Kubernetes bootstrap resources

cert-manager must go before KEDA to avoid blocking on finalizers, and KEDA must go after deleting any `ScaledObjects`.

```bash
# Delete ScaledObjects first to clear KEDA finalizers
kubectl delete scaledobjects --all -n langsmith 2>/dev/null || true

helm uninstall cert-manager -n cert-manager
kubectl delete namespace cert-manager

# KEDA — only installed when enable_langsmith_deployment = true
helm uninstall keda -n keda
kubectl delete namespace keda

helm uninstall envoy-gateway -n envoy-gateway-system
kubectl delete namespace envoy-gateway-system
```

<Note>
When you uninstall Envoy Gateway, GCP releases the external IP. If you re-deploy later, a new IP is issued and you must update your DNS A record. Uninstall to reinstall cannot preserve the same IP unless you pre-allocate a static regional address and bind it to the Gateway (not currently wired in this stack).
</Note>

### A4 — Handle KEDA finalizers (stuck namespaces)

If the `langsmith` namespace gets stuck in `Terminating`, KEDA `ScaledObject` finalizers are likely the cause. The KEDA controller is already gone so it cannot clear them:

```bash
for obj in $(kubectl get scaledobjects -n langsmith -o name 2>/dev/null); do
  kubectl patch "$obj" -n langsmith --type=merge -p '{"metadata":{"finalizers":null}}'
done
```

### A5 — Disable deletion protection

Two `tfvars` must be set to `false` before `terraform destroy` succeeds on GKE and Cloud SQL:

```hcl
gke_deletion_protection      = false
postgres_deletion_protection = false
```

Apply the change with targeted modules to avoid reconciling in-cluster add-ons that were just removed:

```bash
cd terraform/modules/gcp
terraform -chdir=infra apply \
  -target=module.gke_cluster \
  -target=module.cloudsql
```

<Note>
A full `make apply` here can re-run Kubernetes and Helm bootstrap paths and recreate components you just removed. Use targeted applies.
</Note>

### A6 — Destroy GCP infrastructure

```bash
cd terraform/modules/gcp
source infra/scripts/setup-env.sh
make destroy
```

Terraform destroys in dependency order: k8s-bootstrap, Cloud SQL, Memorystore, GCS (only when `storage_force_destroy = true` or the bucket is empty), Workload Identity service account + IAM bindings, GKE cluster and node pools, VPC, subnet, Cloud Router, Cloud NAT.

<Note>
Terraform needs `TF_VAR_postgres_password` even during destroy for provider validation. If the Secret Manager secret no longer exists, set it manually: `export TF_VAR_postgres_password="any-placeholder"`.
</Note>

### A7 — Clean up Secret Manager secrets

If `enable_secret_manager_module = true` was set, the Secret Manager secrets are destroyed by Terraform. If additional secrets were stored manually via `setup-env.sh`, remove them:

```bash
PROJECT_ID="<your-project-id>"
PREFIX="<name_prefix>-<environment>"

gcloud secrets list --project "$PROJECT_ID" --filter="name~langsmith"

gcloud secrets delete "${PREFIX}-postgres-password"             --project "$PROJECT_ID" --quiet
gcloud secrets delete "${PREFIX}-langsmith-license-key"         --project "$PROJECT_ID" --quiet
gcloud secrets delete "${PREFIX}-langsmith-jwt-secret"          --project "$PROJECT_ID" --quiet
gcloud secrets delete "${PREFIX}-langsmith-api-key-salt"        --project "$PROJECT_ID" --quiet
gcloud secrets delete "${PREFIX}-langsmith-admin-password"      --project "$PROJECT_ID" --quiet
gcloud secrets delete "${PREFIX}-deployments-encryption-key"    --project "$PROJECT_ID" --quiet 2>/dev/null || true
gcloud secrets delete "${PREFIX}-agent-builder-encryption-key"  --project "$PROJECT_ID" --quiet 2>/dev/null || true
gcloud secrets delete "${PREFIX}-insights-encryption-key"       --project "$PROJECT_ID" --quiet 2>/dev/null || true
```

### A8 — Verify cleanup

```bash
PROJECT_ID="<your-project-id>"
REGION="<region>"
PREFIX="<name_prefix>-<environment>"

gcloud container clusters list --project "$PROJECT_ID"
gcloud sql instances list --project "$PROJECT_ID"
gcloud redis instances list --region "$REGION" --project "$PROJECT_ID"
gsutil ls 2>/dev/null | grep "$PREFIX" || echo "No matching buckets"
gcloud compute networks list --project "$PROJECT_ID" --filter="name~$PREFIX"
gcloud iam service-accounts list --project "$PROJECT_ID" --filter="email~$PREFIX"
gcloud secrets list --project "$PROJECT_ID" --filter="name~langsmith"
```

## Option B — Teardown without Terraform state

Use this when Terraform state is lost (deleted, corrupted, or never configured with a remote backend). Everything must be deleted manually via gcloud CLI in reverse-dependency order.

<Tip>
State loss usually happens with a local backend (`terraform.tfstate` on disk) when the file is deleted during a directory restructure, or when a remote GCS backend was never configured. Configure a GCS backend on day one to avoid this entire scenario.
</Tip>

### B0 — Inventory what exists

```bash
PROJECT_ID="<your-project-id>"
REGION="<region>"
PREFIX="<name_prefix>-<environment>"

echo "=== GKE ===" && gcloud container clusters list --project "$PROJECT_ID"
echo "=== Cloud SQL ===" && gcloud sql instances list --project "$PROJECT_ID"
echo "=== Memorystore ===" && gcloud redis instances list --region "$REGION" --project "$PROJECT_ID"
echo "=== GCS ===" && gsutil ls 2>/dev/null | grep "$PREFIX"
echo "=== VPC ===" && gcloud compute networks list --project "$PROJECT_ID" --filter="name~$PREFIX"
echo "=== Service Accounts ===" && gcloud iam service-accounts list --project "$PROJECT_ID" --filter="email~$PREFIX"
echo "=== Secret Manager ===" && gcloud secrets list --project "$PROJECT_ID" --filter="name~langsmith"
```

### B1 — Remove Kubernetes resources

```bash
gcloud container clusters get-credentials "$PREFIX-gke-<suffix>" \
  --region "$REGION" --project "$PROJECT_ID"

# Delete LGP CRD (retained by resource policy)
kubectl delete crd lgps.apps.langchain.ai 2>/dev/null || true

# Delete ScaledObjects before KEDA (clears finalizers)
kubectl delete scaledobjects --all -A 2>/dev/null || true

helm uninstall langsmith       -n langsmith              2>/dev/null || true
helm uninstall cert-manager    -n cert-manager           2>/dev/null || true
helm uninstall keda            -n keda                   2>/dev/null || true
helm uninstall envoy-gateway   -n envoy-gateway-system   2>/dev/null || true

kubectl delete namespace langsmith cert-manager keda envoy-gateway-system 2>/dev/null || true
```

**Stuck namespaces — KEDA finalizers:**

```bash
for obj in $(kubectl get scaledobjects -n langsmith -o name 2>/dev/null); do
  kubectl patch "$obj" -n langsmith --type=merge -p '{"metadata":{"finalizers":null}}'
done
```

### B2 — Delete the GKE cluster

```bash
gcloud container clusters list --project "$PROJECT_ID"

gcloud container clusters delete "$PREFIX-gke-<suffix>" \
  --region "$REGION" --project "$PROJECT_ID" --quiet
```

<Note>
GKE cluster deletion takes ~5 minutes and automatically releases the external IP used by the Envoy Gateway.
</Note>

### B3 — Delete the Cloud SQL instance

```bash
gcloud sql instances describe "$PREFIX-pg-<suffix>" \
  --project "$PROJECT_ID" --format="value(settings.deletionProtectionEnabled)"

gcloud sql instances patch "$PREFIX-pg-<suffix>" \
  --project "$PROJECT_ID" --no-deletion-protection

gcloud sql instances delete "$PREFIX-pg-<suffix>" \
  --project "$PROJECT_ID" --quiet
```

### B4 — Delete the Memorystore instance

```bash
gcloud redis instances delete "$PREFIX-redis-<suffix>" \
  --region "$REGION" --project "$PROJECT_ID" --quiet
```

### B5 — Empty and delete the GCS bucket

```bash
gsutil -m rm -r gs://"$PREFIX-traces-<suffix>"
gsutil rb gs://"$PREFIX-traces-<suffix>"
```

If the bucket has versioned objects:

```bash
gsutil -m rm -a gs://"$PREFIX-traces-<suffix>"/**
gsutil rb gs://"$PREFIX-traces-<suffix>"
```

### B6 — Delete Secret Manager secrets

```bash
for secret in $(gcloud secrets list --project "$PROJECT_ID" \
  --filter="name~langsmith" --format="value(name)"); do
  gcloud secrets delete "$secret" --project "$PROJECT_ID" --quiet
done
```

### B7 — Delete the Workload Identity service account

```bash
gcloud iam service-accounts list --project "$PROJECT_ID" --filter="email~$PREFIX"

gcloud iam service-accounts delete "$PREFIX-langsmith@$PROJECT_ID.iam.gserviceaccount.com" \
  --project "$PROJECT_ID" --quiet
```

### B8 — Delete VPC and networking

<Warning>
Must be done last. Subnets cannot be deleted while GKE is still running.
</Warning>

```bash
VPC_NAME="$PREFIX-vpc-<suffix>"

# 1. Cloud NAT
gcloud compute routers nats delete "$PREFIX-nat-<suffix>" \
  --router="$PREFIX-router-<suffix>" \
  --region="$REGION" --project "$PROJECT_ID" --quiet

# 2. Cloud Router
gcloud compute routers delete "$PREFIX-router-<suffix>" \
  --region="$REGION" --project "$PROJECT_ID" --quiet

# 3. Subnets
for subnet in $(gcloud compute networks subnets list \
  --network="$VPC_NAME" --project "$PROJECT_ID" --format="value(name)"); do
  gcloud compute networks subnets delete "$subnet" \
    --region="$REGION" --project "$PROJECT_ID" --quiet
done

# 4. Private service connection (VPC peering for Cloud SQL / Memorystore)
gcloud services vpc-peerings delete \
  --network="$VPC_NAME" --project "$PROJECT_ID" --quiet 2>/dev/null || true

# 5. VPC
gcloud compute networks delete "$VPC_NAME" --project "$PROJECT_ID" --quiet
```

<Note>
If VPC deletion fails with `"has active peerings"`, the private service peering for Cloud SQL or Memorystore is still attached. The `gcloud services vpc-peerings delete` command above handles it. If that fails, wait ~2 minutes for Cloud SQL deletion to propagate, then retry.
</Note>

### B9 — Verify cleanup

```bash
gcloud container clusters list --project "$PROJECT_ID" | grep "$PREFIX" || echo "GKE: clean"
gcloud sql instances list --project "$PROJECT_ID" | grep "$PREFIX" || echo "Cloud SQL: clean"
gcloud redis instances list --region "$REGION" --project "$PROJECT_ID" | grep "$PREFIX" || echo "Redis: clean"
gsutil ls 2>/dev/null | grep "$PREFIX" || echo "GCS: clean"
gcloud compute networks list --project "$PROJECT_ID" --filter="name~$PREFIX" | grep "$PREFIX" || echo "VPC: clean"
gcloud iam service-accounts list --project "$PROJECT_ID" --filter="email~$PREFIX" | grep "$PREFIX" || echo "IAM SA: clean"
gcloud secrets list --project "$PROJECT_ID" --filter="name~langsmith" | grep langsmith || echo "Secrets: clean"
```

## Parallelization

| Can run in parallel | Wait required before |
|---|---|
| Cloud SQL, Memorystore, GCS, Secret Manager | Independent — start all at once after GKE is deleted |
| GKE cluster | Must complete before VPC deletion |
| Cloud NAT + Router | Must complete before subnet deletion |
| Subnet deletion | Must complete before VPC deletion |

## Lessons learned

- Configure a remote backend (GCS) before `terraform apply`. Local state is fragile and easily lost.
- KEDA finalizers block namespace deletion when the KEDA controller is uninstalled first. Delete `ScaledObjects` before uninstalling KEDA, or patch out the finalizers.
- The LGP CRD is kept by resource policy. `helm uninstall` does not remove it; delete manually with `kubectl delete crd lgps.apps.langchain.ai`.
- GKE deletion releases the external IP. Re-deploy issues a new IP. Update your DNS A record after re-deploy. Use a static regional IP to avoid this (not currently wired in the stack).
- Private service connection peering (`servicenetworking-googleapis-com`) must be removed before the VPC can be deleted. The `gcloud services vpc-peerings delete` command handles it.
- Cloud SQL deletion takes ~2 minutes. The VPC peering is not released until the instance is fully gone. Wait before VPC cleanup.
- GCS buckets with versioned objects require `gsutil rm -a` (all versions) before `gsutil rb` succeeds.
